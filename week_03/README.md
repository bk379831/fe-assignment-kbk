# fe-assignment-kbk
## 0. 문제 인식
### 1. 클래스 이름만 보고 역할을 명확하게 알 수 있는가?
아니다.
.box1, .box2, .red-text, .big-text와 같은 클래스 이름은 요소의 구조나 의미가 아니라 단순한 스타일만 나타낸다.
따라서 해당 요소가 어떤 역할을 하는지 직관적으로 파악하기 어렵고, 코드의 가독성이 떨어진다.
### 2. 스타일 충돌(겹침)이 발생할 가능성은 없는가?
여러 클래스(red-text, big-text)를 동시에 적용하는 방식은 서로 다른 스타일이 겹치면서 예상하지 못한 결과를 만들 수 있다.
특히 프로젝트가 커질수록 동일한 클래스가 여러 곳에서 재사용되면서 스타일 충돌 및 덮어쓰기 문제가 발생할 가능성이 높다.
### 3. 프로젝트가 커졌을 때 유지보수는 쉬울까?
어렵다.
이 방식은 스타일 기준으로 클래스가 나뉘어 있기 때문에 특정 UI 요소를 수정하려고 할 때 관련된 모든 클래스를 찾아야 한다.
또한 클래스 이름만으로 구조를 파악하기 힘들어 코드 수정 시 시간이 오래 걸리고 실수 가능성이 증가한다.
### 4. 이런 방식이 실무에서 문제가 되는 이유
이 방식은 다음과 같은 이유로 실무에서 문제가 된다.

- 클래스 이름이 의미를 담고 있지 않아 코드 이해가 어렵다
- 스타일이 분산되어 있어 관리가 비효율적이다
- 여러 클래스 조합으로 인해 스타일 충돌이 발생하기 쉽다
- UI 구조와 스타일이 분리되지 않아 유지보수가 어렵다

결과적으로 협업 환경에서는 코드 일관성이 깨지고, 유지보수 비용이 증가하는 문제가 발생한다.

## 1. 조사 및 정리
### 1. CSS를 무작정 작성했을 때 생기는 문제
- 클래스 이름이 직관적이지 않다.
- 스타일 중복이 발생할 수 있다.
- 유지보수가 어렵다.  
- 코드 가독성이 저하된다.
### 2. BEM이란 무엇인가가
BEM은 Block, Element, Modifier의 약자이다. UI를 독립적인 블록 단위로 나누고 구조적으로 클래스 이름을 설계하는 방법이다.
- 예시:
    ```<div class="profile-card">
    <h1 class="profile-card__name">이름</h1>
    <button class="profile-card__button profile-card__button--primary">버튼</button>
    </div>
### 3. Tailwind CSS는 어떤 방식인가
Tailwind CSS는 Utility-first 방식이다. 
- Utility-first 방식: 작은 기능 단위의 클래스를 조합하여 HTML에서 직접 스타일을 적용하는 방식이다.

## 2. UI 구조 설계
1) 카드 전체 영역
- 전체 UI를 감싸는 컨테이너
- 중앙 정렬 및 배경 스타일 적용
2) 프로필 이미지 영역
- 사용자의 프로필 이미지를 표시하는 영역
- 원형 형태로 디자인하여 시각적으로 강조
3) 텍스트 영역
- 이름 영역
    사용자의 이름을 크게 강조하여 표시
- 한 줄 소개 영역
    간단한 자기소개 문장을 표시
4) 기술 태그 영역
- 사용자의 관심 기술을 태그 형태로 표시
- 여러 개의 태그를 나열하여 가독성을 높임
5) 버튼 영역
- 외부 링크로 이동하는 버튼 배치
    GitHub 버튼
    Blog / Portfolio 버튼
### 설계 의도
- 정보를 이미지 → 텍스트 → 태그 → 버튼 순서로 배치하여 사용자가 자연스럽게 내용을 이해할 수 있도록 구성하였다.
- 각 영역을 명확하게 구분하여 이후 BEM 구조 설계 및 스타일 적용이 용이하도록 하였다.
